# Framework Upgrade Guide (Admin)

This document is for the Dubhe framework admin — the address stored in
`DappHub.config.admin` — who is responsible for shipping framework upgrades
and managing framework-level governance objects.

---

## Framework Upgrade Architecture

```
framework/src/dubhe/
├── sources/
│   ├── systems/dapp_system.move      ← public API; contains FRAMEWORK_VERSION
│   ├── core/dapp_service.move        ← DappHub storage layer; holds version field
│   └── scripts/
│       ├── deploy_hook.move          ← runs once on first publish (init-time)
│       └── migrate.move              ← ON_CHAIN_VERSION + on_chain_version()
└── sources/codegen/
    └── genesis.move                  ← AUTO-GENERATED; migrate() body is an
                                         extension point, currently empty
```

### Version constants

| Constant            | Type  | File                | Purpose                                                                         |
| ------------------- | ----- | ------------------- | ------------------------------------------------------------------------------- |
| `FRAMEWORK_VERSION` | `u64` | `dapp_system.move`  | Compile-time constant embedded in every framework package binary                |
| `DappHub.version`   | `u64` | `dapp_service.move` | On-chain state; bumped by `dapp_system::bump_framework_version` during upgrades |

## Functions Gated by `assert_framework_version`

These `dapp_system` functions check that the calling package's `FRAMEWORK_VERSION`
matches `DappHub.version` before executing:

| Function               | Effect if version mismatch       |
| ---------------------- | -------------------------------- |
| `create_dapp`          | `not_latest_version_error` abort |
| `create_user_storage`  | `not_latest_version_error` abort |
| `settle_writes`        | `not_latest_version_error` abort |
| `suspend_dapp`         | `not_latest_version_error` abort |
| `unsuspend_dapp`       | `not_latest_version_error` abort |
| `update_framework_fee` | `not_latest_version_error` abort |

Functions **not** gated (must always be accessible): `get_field`, `has_record`,
`ensure_latest_version`, `ensure_not_paused`, `propose_ownership`, `accept_ownership`,
`set_metadata`, `upgrade_dapp`, `recharge_credit`.

## Upgrade Types

### Compatible Framework Upgrade (no version bump)

Add or fix code without changing `FRAMEWORK_VERSION`. Run:

```sh
dubhe upgrade --name dubhe --network testnet
```

`upgradeHandler` detects no pending schema changes and skips the migration
transaction. `DappHub.version` stays the same. All DApps remain operational.

### Breaking Framework Upgrade (FRAMEWORK_VERSION bump)

When a breaking protocol change requires all old framework packages to be
invalidated:

1. Increment `FRAMEWORK_VERSION` in `dapp_system.move`.
2. If new resources are added, update `dubhe.config.ts` and run `dubhe generate`
   to regenerate `genesis.move`.
3. Run `dubhe upgrade --name dubhe --network testnet`.

After the upgrade, `upgradeHandler` detects pending schema changes, generates a
`migrate_to_vN` function in `migrate.move`, and submits a second on-chain
transaction calling:

```
<newPackageId>::migrate::migrate_to_vN(dapp_hub, dapp_storage, new_package_id, new_version)
```

`migrate_to_vN` calls `dapp_system::upgrade_dapp` which:

- Appends the new package ID to `dapp_storage.package_ids`.
- Bumps `dapp_storage.version` to the new version.

`genesis::migrate` is also called by `migrate_to_vN` as an extension point;
its body is currently empty and is regenerated by `dubhe generate`.

If there are **no** new resources, the migration transaction is skipped entirely.

## Step-by-Step Framework Upgrade Procedure

### 1. (If breaking) Increment `FRAMEWORK_VERSION`

```move
// framework/src/dubhe/sources/systems/dapp_system.move
const FRAMEWORK_VERSION: u64 = 2;  // was 1
```

### 2. (If new resources) Update `dubhe.config.ts` and run generate

```sh
# From the framework/ directory
node node_modules/@0xobelisk/sui-cli/dist/dubhe.js generate
```

This regenerates `sources/codegen/genesis.move`.

### 3. Run the upgrade

```sh
dubhe upgrade --name dubhe --network testnet
```

`upgradeHandler`:

1. Builds the new framework package.
2. Submits the `authorize_upgrade` + `upgrade` + `commit_upgrade` PTB.
3. If new resources were detected, waits 5 seconds, then submits a second
   transaction calling `migrate_to_vN`.
4. Saves the new `packageId` and `version` to `.history/sui_<network>/latest.json`.

### 4. Verify on-chain state

```sh
cat framework/src/dubhe/Published.toml
cat framework/src/dubhe/.history/sui_testnet/latest.json
```

Confirm `version` incremented and the new package ID is recorded.

## Framework Admin Governance

### Two-step admin transfer

`propose_framework_admin` / `accept_framework_admin` implement Ownable2Step:

1. Current admin nominates a new admin address.
2. Nominee calls `accept_framework_admin` to complete the transfer.
3. Proposing `@0x0` cancels the pending nomination.

Only the current admin can call `propose_framework_admin`.
Only the nominated address can call `accept_framework_admin`.

### Framework admin vs treasury

| Role            | Address stored in             | Key permissions                                                                                          |
| --------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------- |
| Framework admin | `DappHub.config.admin`        | `suspend_dapp`, `unsuspend_dapp`, `update_framework_fee`, `propose_framework_admin`, `grant_free_credit` |
| Treasury        | `DappHub.fee_config.treasury` | `propose_treasury` / `accept_treasury` (receives settlement payments)                                    |

The treasury address **cannot** call `suspend_dapp`, `unsuspend_dapp`, or
`update_framework_fee`. This separation prevents a compromised treasury key
from affecting DApp availability.

### Fee updates

```move
// new_base_fee and new_bytes_fee are both u256.
// Decrease: takes effect immediately.
// Increase: subject to a 48-hour delay (MIN_FEE_INCREASE_DELAY_MS).
dapp_system::update_framework_fee(dapp_hub, new_base_fee, new_bytes_fee, clock, ctx);
```

`update_framework_fee` is version-gated — the caller must use the current
framework package.

## Security Boundaries

### What happens if a breaking upgrade has no migration

If `FRAMEWORK_VERSION` was bumped but no new resources were added,
`upgradeHandler` skips the migration transaction. `DappHub.version` is **not**
automatically advanced. Version-gated functions from the new package will abort
until `bump_framework_version` is called (typically via a custom migration path).

### Idempotency of `bump_framework_version`

```move
public(package) fun bump_framework_version(dh: &mut DappHub) {
    let current = dapp_service::framework_version(dh);
    if (FRAMEWORK_VERSION > current) {
        dapp_service::set_framework_version(dh, FRAMEWORK_VERSION);
    };
}
```

Calling it multiple times is safe; the second call is a no-op.

### DApp impact during framework upgrade

Between the upgrade transaction and the migration transaction:

- New framework package cannot call version-gated functions (version mismatch).
- Old framework package continues to work (version still matches hub).
- DApps using the old framework package are unaffected.

The 5-second delay in `upgradeHandler` mitigates indexer lag on localnet.

### Epoch-delayed fee increases

When `update_framework_fee` increases the fee, it stores a `PendingFee` with
a future `effective_at` timestamp (`now + MIN_FEE_INCREASE_DELAY_MS = 48 h`).
`get_effective_fees_at` returns the current fee until that timestamp passes,
giving DApp operators time to recharge credits.
