# Technology Deep-Dive

## Overview

As of **March 29, 2026**, Dubhe is documented and productized as a **Sui-first** stack with three major layers:

1. **Dubhe Engine**: schema-driven Move development and codegen workflow
2. **Dubhe Channel**: off-chain realtime execution/query plane over HTTP + SSE
3. **Dubhe OS**: governance/coordination direction

## System Layers

| Layer            | Responsibility                                 | Current State       |
| ---------------- | ---------------------------------------------- | ------------------- |
| Application      | dApps, game clients, backend services          | Implemented         |
| SDK/Tooling      | CLI, TS SDK, indexer, GraphQL stack            | Implemented         |
| Channel Runtime  | submit/query/nonce/subscription + ops controls | Implemented         |
| On-chain Runtime | Sui Move execution and finality                | Implemented         |
| Governance Layer | extended coordination and policy surface       | Directional roadmap |

## Dubhe Channel: Implemented Interface Model

### Public Routes

- `GET /health`
- `GET /ready`
- `GET /metrics`
- `POST /submit`, `POST /submit_batch`
- `POST /nonce`
- `POST /get_table`
- `POST /subscribe_table` (SSE)
- `POST /v2/query`, `POST /v2/subscribe`, `POST /v2/publish`

### Sui SDK Mapping

In `@0xobelisk/sui-client`:

- `latestNonce` -> `/nonce`
- `submitToChannel` -> `/submit`
- `queryChannelTable` -> `/get_table`
- `subscribeChannelTable` -> `/subscribe_table`

This gives a practical path for apps that need low-latency read/write while keeping Sui as the source of truth for on-chain state.

## Runtime Governance and Reliability

### Admission and Limits

Channel includes request admission and SSE connection governance:

- per-client inflight controls for submit/publish/subscribe
- SSE global/per-tenant/per-client connection limits
- optional lifetime, idle timeout, and slow-consumer disconnect thresholds

### Tenant Isolation

Hosted mode supports deployment scoping by host/header/default deployment settings to isolate tenants in shared infrastructure.

### Operational Endpoints

- `/ready` reports runtime readiness and operational gate state
- `/metrics` exports Prometheus-style metrics for production monitoring

## Performance Interpretation

Dubhe Channel is designed for low-latency interaction, but concrete latency and throughput vary by:

- hardware and network profile
- backend mode (in-memory vs Redis+NATS)
- configured governance limits
- workload shape

Public benchmark numbers should be accompanied by reproducible methodology and environment details.

## Design Choices and Tradeoffs

| Decision                           | Benefit                                          | Tradeoff                                           |
| ---------------------------------- | ------------------------------------------------ | -------------------------------------------------- |
| Sui-first scope                    | sharper product quality and faster iteration     | slower near-term ecosystem breadth                 |
| HTTP + SSE focus                   | simple integration and operational clarity       | less protocol variety than multi-transport designs |
| Strict nonce + admission controls  | predictable runtime safety under load            | stricter client sequencing requirements            |
| v1 SDK mapping + v2 runtime routes | stable client UX with extensible backend surface | dual-route documentation complexity                |

## Implemented vs Directional

### Implemented Today

- Sui-first development workflow
- Channel v1+v2 HTTP/SSE runtime surface
- readiness/metrics and governance controls

### Directional

- wider non-Sui ecosystem support
- deeper verification model enhancements
- expanded enterprise operations features
