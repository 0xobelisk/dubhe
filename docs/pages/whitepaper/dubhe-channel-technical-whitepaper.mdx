# Dubhe Channel Technical Whitepaper

## Status Snapshot (March 29, 2026)

This document describes Dubhe Channel based on the current implementation in:

- `crates/dubhe-channel/src/main.rs`
- `packages/sui-client/src/libs/dubheChannel/client.ts`
- `packages/sui-client/src/dubhe.ts`

It separates **implemented behavior** from **roadmap direction** to keep technical and product claims consistent.

## 1. Positioning

Dubhe Channel is the off-chain realtime execution and data path used by Dubhe applications. In its current form, it provides:

- low-latency write submission (`/submit`, `/submit_batch`)
- nonce management for ordered execution (`/nonce`)
- table reads (`/get_table`, `/v2/query`)
- realtime subscriptions over SSE (`/subscribe_table`, `/v2/subscribe`)
- operational health and metrics (`/health`, `/ready`, `/metrics`)

For Sui developers, the default integration path is through `@0xobelisk/sui-client` channel helpers.

## 2. Runtime Architecture (Implemented)

### Core Runtime Path

1. Client submits request through HTTP.
2. Channel validates request shape, tenant scope, and admission limits.
3. Execution/query path runs against in-memory state or configured runtime backends.
4. Materialized table updates are stored and broadcast through SSE streams.
5. Metrics and readiness states are exposed for operations.

### Backends

Current runtime supports:

- in-memory runtime (default)
- Redis + NATS backed mode (`redis_url` + `nats_url`)

### API Protocols

Implemented today:

- HTTP JSON endpoints
- SSE streams (`text/event-stream`) for subscriptions

Not implemented as public API in the current server:

- gRPC public interface
- Ethereum JSON-RPC compatibility layer

## 3. API Surface (Implemented)

| Route                               | Method | Purpose                    |
| ----------------------------------- | ------ | -------------------------- |
| `/health`                           | `GET`  | Liveness                   |
| `/ready`                            | `GET`  | Readiness and ops snapshot |
| `/metrics`                          | `GET`  | Prometheus metrics         |
| `/submit`, `/v2/submit`             | `POST` | Single request submit      |
| `/submit_batch`, `/v2/submit_batch` | `POST` | Batch submit               |
| `/nonce`, `/v2/nonce`               | `POST` | Next nonce by sender       |
| `/get_table`                        | `POST` | Table value query          |
| `/subscribe_table`                  | `POST` | Table SSE subscription     |
| `/v2/query`                         | `POST` | Runtime snapshot query     |
| `/v2/subscribe`                     | `POST` | Runtime event subscription |
| `/v2/publish`                       | `POST` | Runtime event publish      |

### Important Semantics

- `submit` requires strict nonce ordering (`expected == submitted`).
- `subscribe_table` request fields are optional in schema, but matching logic requires `dapp_key`.
- `ready` returns `200` when healthy, `503` when not ready.
- `metrics` exposes Prometheus-formatted counters/gauges for submit/query/nonce/SSE/materialization/ops signals.

## 4. Multi-Tenant Isolation and Routing

Channel supports hosted multi-tenant mode:

- `hosted_mode=true` enables tenant scoping.
- Tenant can be resolved from host subdomain, deployment headers, or default deployment id.
- Headers used: `x-obelisk-deployment`, `x-obelisk-deployment-slug`.
- Header override behavior is gated by `allow_deployment_header_override`.

In hosted mode, non-health routes require a resolvable deployment scope.

## 5. Realtime Governance and Fault Handling

### Request Admission

Per-client inflight controls are configurable for:

- submit
- publish
- subscribe

When limits are exceeded, requests are rejected with `429`.

### SSE Governance

SSE connection admission can enforce:

- global connection cap
- per-tenant connection cap
- per-client connection cap
- max connection lifetime
- idle timeout
- lagged message threshold (slow consumer protection)

### Realtime Gate Controls

Runtime gates can deny new realtime streams and disconnect active streams when required by operations.

## 6. Security Model (Current vs Planned)

### Implemented

- tenant scoping in hosted mode
- deterministic nonce enforcement per sender scope
- request admission throttles
- request id propagation (`x-request-id`)
- ops/admin token guards for protected operations endpoints

### Planned / Future Direction

- TEE-based execution attestation
- fraud-proof style dispute pipelines
- broader cross-chain execution unification

These are roadmap directions and should not be treated as generally available production features unless explicitly released.

## 7. Performance Statement

As of March 29, 2026, this repository does not publish an audited benchmark report that proves fixed public numbers (for example, guaranteed TPS/latency targets across environments).

Performance depends on:

- runtime backend mode (in-memory vs Redis+NATS)
- hardware/network profile
- governance limits and traffic shape

Any target metrics should be treated as planning goals until accompanied by reproducible benchmark artifacts.

## 8. Sui SDK Integration

The Sui client currently maps channel helpers to v1 endpoints:

- `latestNonce` -> `/nonce`
- `submitToChannel` -> `/submit`
- `queryChannelTable` -> `/get_table`
- `subscribeChannelTable` -> `/subscribe_table`

Reference guide:

- [Dubhe Channel (Sui)](/dubhe/sui/channel)

## 9. Roadmap Summary

Near-term priorities:

- improve operational hardening for production rollouts
- keep v2 runtime API stable for richer event/query scenarios
- continue Sui-first productization with explicit compatibility boundaries

Longer-term directions:

- cross-chain expansion
- stronger verification model
- enterprise deployment controls

Roadmap items are directional and may shift with product priorities.
