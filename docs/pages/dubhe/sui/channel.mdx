# Dubhe Channel

This page is aligned to the current server implementation in:

- `crates/dubhe-channel/src/main.rs`
- `packages/sui-client/src/dubhe.ts`
- `packages/sui-client/src/libs/dubheChannel/client.ts`

## API Surface (Implemented)

| Route                               | Method | Purpose                                                  | Response type         |
| ----------------------------------- | ------ | -------------------------------------------------------- | --------------------- |
| `/health`                           | `GET`  | Liveness check                                           | JSON                  |
| `/ready`                            | `GET`  | Readiness snapshot (gates/limits/materialization health) | JSON (`200` or `503`) |
| `/metrics`                          | `GET`  | Prometheus-style metrics                                 | `text/plain`          |
| `/submit`, `/v2/submit`             | `POST` | Submit one off-chain execution request                   | JSON                  |
| `/submit_batch`, `/v2/submit_batch` | `POST` | Submit a batch of execution requests                     | JSON                  |
| `/nonce`, `/v2/nonce`               | `POST` | Read next nonce for `sender`                             | JSON                  |
| `/get_table`                        | `POST` | Read materialized table value by key                     | JSON                  |
| `/subscribe_table`                  | `POST` | Subscribe table updates via SSE                          | `text/event-stream`   |
| `/v2/query`                         | `POST` | Generic runtime snapshot query (`SnapshotQuery`)         | JSON                  |
| `/v2/subscribe`                     | `POST` | Generic runtime subscription (`SubscriptionSpec`)        | `text/event-stream`   |
| `/v2/publish`                       | `POST` | Publish runtime event (`EventEnvelope`)                  | JSON                  |

## Sui SDK Mapping

In `Dubhe` (`@0xobelisk/sui-client`), the channel helpers map to v1 routes:

- `latestNonce` -> `POST /nonce`
- `submitToChannel` -> `POST /submit`
- `queryChannelTable` -> `POST /get_table`
- `subscribeChannelTable` -> `POST /subscribe_table`

`/v2/*` endpoints are available for advanced runtime integrations and are not required for standard Sui client usage.

## Route Semantics You Should Know

### `POST /nonce`

Request:

```json
{ "sender": "0x..." }
```

Response:

```json
{ "nonce": 123 }
```

The server returns the **next expected nonce** for this account scope.

### `POST /submit`

Request shape:

```json
{
  "chain": "sui",
  "sender": "0x...",
  "nonce": 123,
  "ptb": { "version": 2, "inputs": [], "commands": [] },
  "signature": "..."
}
```

Important validation behavior:

- `nonce` is required and must equal server-side next nonce.
- `chain` currently accepts `sui` (SDK default); server also has `evm`/`solana` branches.
- Nonce mismatch returns `400`.

### `POST /get_table`

Request shape:

```json
{
  "dapp_key": "<package>::dapp_key::DappKey",
  "account": "<account_without_0x>",
  "table": "position",
  "key": []
}
```

Response shape:

```json
{
  "message": true,
  "data": []
}
```

### `POST /subscribe_table` (SSE)

Request fields are optional in the schema:

```json
{
  "dapp_key": "<package>::dapp_key::DappKey",
  "account": "...",
  "table": "position",
  "key": []
}
```

But matching logic requires `dapp_key` in practice:

- `dapp_key` missing => no event match
- then optional narrowing by `account`, `table`, `key`

SSE stream behavior:

- sends currently matched initial data first
- sends heartbeats (`: heartbeat`) periodically
- can disconnect on max lifetime, idle timeout, slow consumer, realtime gate disabled, or upstream close

## Multi-Tenant / Hosted Mode

When `hosted_mode=false`, requests run in single-tenant mode.

When `hosted_mode=true`, tenant scope is resolved from:

- host subdomain (with `--hosted-domain-suffix`)
- headers `x-obelisk-deployment` / `x-obelisk-deployment-slug`
- `--default-deployment-id` fallback

If `allow_deployment_header_override=true`, deployment headers override host-derived scope.

In hosted mode, requests (except `/health`, `/ready`, `/metrics`) require resolvable deployment scope.

## Admission Control and Runtime Governance

Client identity is resolved in this order:

- `x-forwarded-for`
- `x-real-ip`
- socket remote address

Per-client inflight limits can reject requests with `429`:

- `http_max_submit_inflight_per_client`
- `http_max_publish_inflight_per_client`
- `http_max_subscribe_inflight_per_client`

SSE connection limits can also reject with `429`:

- global / per-tenant / per-client limits
- optional max lifetime, idle timeout, lagged-message threshold

## Quick cURL Checks

```bash
curl -s http://localhost:8080/health
```

```bash
curl -s -X POST http://localhost:8080/nonce \
  -H 'Content-Type: application/json' \
  -d '{"sender":"0x15fde77101778fafe8382743171294dfd8e7900a547711ee375379c27a85fd31"}'
```

```bash
curl -N -X POST http://localhost:8080/subscribe_table \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  -d '{"dapp_key":"<package>::dapp_key::DappKey","table":"position"}'
```
