# Why Sui Move

Dubhe currently prioritizes Sui Move because it gives the best balance of safety, performance, and developer workflow for real-time on-chain applications.

## Key reasons

### 1) Object model matches game/world state

Sui's object model maps naturally to entity-heavy applications:

- characters
- inventories
- world resources
- capability-guarded operations

This reduces accidental complexity when modeling shared world state.

### 2) Parallelism potential

Independent objects can be executed concurrently, which improves throughput for multi-user interactive flows.

For Dubhe use cases (real-time dApps, game-like state transitions), this is a strong practical advantage.

### 3) Move safety properties

Move's type/resource semantics help prevent common smart-contract bugs:

- accidental asset duplication/destruction
- unsafe ownership transfer patterns
- weak type boundaries in state transitions

This improves default correctness for teams shipping fast.

### 4) End-to-end tooling completeness

Sui has a mature stack Dubhe can compose with:

- CLI + RPC
- GraphQL/gRPC read paths
- solid localnet/testnet workflows

This makes DevEx predictable from local debugging to production deployment.

## Tradeoffs and constraints

- Ecosystem choice: focusing on one chain reduces short-term multi-chain surface area.
- Dependency drift: Sui/tooling versions move fast and docs must keep up.
- Upgrade complexity: package/version upgrades require careful ID/config migration.

See [World Upgrades](/dubhe/sui/world-upgrades) for operational upgrade strategy.

## Positioning

Dubhe is not coupled to a single conceptual model forever, but current production-grade implementation quality is strongest on Sui Move.

For architecture context, see [Architecture](/dubhe/sui/architecture).
